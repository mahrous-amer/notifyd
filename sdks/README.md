# notifyd SDKs

Official client libraries for the notifyd notification delivery API, one per language:

- [`go/`](go) — `github.com/mahrous-amer/notifyd/sdks/go`
- [`typescript/`](typescript) — `@notifyd/sdk`
- [`python/`](python) — `notifyd-sdk`

## Why hand-written, not generated

`docs/openapi.yaml` is the source of truth for the API surface, but these SDKs are
hand-written rather than generated by openapi-generator. `openapi-generator`'s `go`
generator emits a `Configuration.APIKey` struct that unconditionally collides with the
`APIKey` schema (the `/keys` resource) — confirmed across three attempts (plain
generation, `--type-mappings`, `--model-name-suffix`), each failing differently. Fixing
it would mean renaming a public schema to accommodate a code generator's internal
naming, or patching and pinning a forked generator template. Given the API surface is
14 non-admin endpoints, a careful handwritten client per language was the better
tradeoff for v1.

Every language's typed request/response shapes mirror the OpenAPI schemas 1:1 (field
names match the JSON keys), so a future move to generation is a non-breaking internal
swap, not a wrapper API change.

## Common wrapper API surface

All three SDKs expose the same shape, adapted to each language's idioms:

- Constructor: `(apiKey, apiSecret, baseURL?)` — `baseURL` defaults to
  `https://notifyd.fluxintek.com/api`.
- Automatic token exchange (`POST /auth/token`), cached until 30 seconds before expiry,
  with a single forced-refresh retry on any `401`.
- `send(...)`, `sendMulti(...)` — notification delivery.
- `listNotifications`, `getNotification`, `listAttempts`, `getMetrics` — notification
  status.
- `listChannels`, `createChannel`, `getChannel`, `updateChannel`, `deleteChannel` —
  channel config CRUD.
- `listApiKeys`, `createApiKey`, `revokeApiKey` — API key management.
- `listWebhooks`, `createWebhook`, `updateWebhook`, `deleteWebhook` — status-webhook
  endpoint CRUD.
- `verifyWebhookSignature(secret, timestamp, body, signatureHeader)` — validates the
  `X-Notifyd-Signature: sha256=<hex>` header notifyd sends on both the `webhook` channel
  type and status-webhook deliveries.

See each language's README for exact method signatures and naming conventions
(camelCase in TypeScript, snake_case in Go/Python).

## Shared signature test vectors

[`testdata/signature_vectors.json`](testdata/signature_vectors.json) is generated from
`internal/provider.SignHMAC` — the exact function notifyd uses to sign outbound webhook
requests — via [`testdata/regen_vectors.sh`](testdata/regen_vectors.sh). All three SDKs'
test suites verify their signature helper against this one fixture, so a bug in any
language's HMAC implementation shows up as a test failure rather than a silent
cross-language inconsistency.

## Running tests

```
# Go
cd sdks/go && go test ./...

# TypeScript
cd sdks/typescript && npm ci && npm test

# Python
cd sdks/python && python3 -m venv .venv && source .venv/bin/activate && pip install -e ".[dev]" && pytest
```

The Go SDK is a separate module (`sdks/go/go.mod`) so its dependencies never affect the
main notifyd service module — `go test ./...` from the repo root does not build or test
it, by design.
